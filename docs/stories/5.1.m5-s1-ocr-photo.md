## Story
**As a** estudante usando a Tutora ENEM,
**I want** enviar uma foto de uma questão do ENEM e receber correção e explicação normalmente,
**so that** eu não precise digitar o enunciado e as alternativas — basta fotografar a questão e a Tutora já entende e processa.

## Acceptance Criteria
1. [AC1] Aluna envia foto (JPEG ou PNG) via Telegram e o sistema detecta `input_mode == "image"`.
2. [AC2] Sistema detecta que é imagem antes de tentar processar como texto.
3. [AC3] Claude Vision extrai enunciado + 5 alternativas (A–E) da imagem e retorna JSON estruturado.
4. [AC4] Parser monta `QuestionSnapshot` com `content`, `alternatives` (A–E) e `correct_alternative=None` por padrão.
5. [AC5] Fluxo normal continua após montagem do snapshot: `bank_match → socratic/direto → anki → envio`.
6. [AC6] Registro persistido em `submitted_questions` com os novos campos: `source='ocr_photo'`, `ocr_raw_text` (resposta bruta do Claude), `ocr_confidence` (float 0.0–1.0 estimado pelo modelo).
7. [AC7] Se OCR falhar (imagem ilegível / Claude retorna erro ou JSON malformado): bot responde "Não consegui ler a foto. Tira outra ou manda o texto?" e não persiste snapshot inválido.
8. [AC8] Testes com 5 fotos reais do ENEM (diferentes tópicos e qualidades de imagem).
9. [AC9] Caching: se mesma foto enviada 2× (mesmo `file_id` Telegram), resultado OCR é reutilizado sem nova chamada à Claude API.

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| OCR falha em imagens de baixa qualidade (foto escura, torta, texto pequeno) | Alta | Médio | AC7 cobre fallback; testar com 5 fotos reais de qualidades variadas (AC8) |
| Claude retorna JSON malformado ou parcial | Média | Médio | Parser defensivo com `try/except json.JSONDecodeError`; tratar como falha (AC7) |
| Custo Claude Vision por imagem (~$0.001–0.005/imagem) | Baixa | Baixo | Cache por `file_id` (AC9) evita chamadas duplicadas; volume baixo no MVP |
| `file_id` Telegram expira após certo período (não é permanente) | Média | Baixo | TTL cache 7 dias é conservador; para MVP aceitável |
| `app/clients/llm.py` pode não suportar `image` content blocks | Média | Alto | Verificar antes de implementar; pode precisar estender `ClaudeClient` |
| In-memory cache não persiste entre reinicializações do processo | Alta | Baixo | Comportamento esperado no MVP; Redis é extensão futura documentada |

### Integração Claude Vision
- Model: `claude-sonnet-4-6`
- Content block: `{"type": "image", "source": {"type": "base64", "media_type": "image/jpeg", "data": "<base64>"}}`
- Prompt sugerido (JSON-only):
  ```
  Você é um parser de questões do ENEM. Extraia do image:
  {"enunciado": "<texto do enunciado>", "alternativas": {"A":"...","B":"...","C":"...","D":"...","E":"..."}, "confianca": 0.0-1.0}
  Responda APENAS com JSON válido, sem markdown.
  ```
- `confianca` é auto-avaliação do modelo (0.0 = ilegível, 1.0 = perfeito)
- Se Claude não retornar JSON válido → tratar como falha (AC7)

### Download de imagem do Telegram
- `getFile`: `GET https://api.telegram.org/bot{TOKEN}/getFile?file_id={file_id}` → retorna `file_path`
- Download: `GET https://api.telegram.org/file/bot{TOKEN}/{file_path}` → bytes da imagem
- Converter para base64 antes de enviar ao Claude

### Cache de OCR
- Implementação simples: `dict[str, tuple[OcrResult, datetime]]`
- TTL: 7 dias — checar `datetime.now() - timestamp > timedelta(days=7)`
- Chave: `file_id` do Telegram (string única por arquivo enviado)
- Para MVP: in-memory (processo-level). Redis é extensão futura.

### `OcrResult` dataclass sugerida
```python
@dataclass
class OcrResult:
    content: str                  # enunciado extraído
    alternatives: list[QuestionAlternative]  # A–E
    ocr_raw_text: str             # JSON bruto retornado pelo Claude
    ocr_confidence: float         # 0.0–1.0
```

### Observações de integração
- `QuestionSnapshot` montado via OCR deve ter `source_mode='ocr_photo'`, `source_truth='ocr_vision'`
- `correct_alternative=None` sempre (OCR não sabe o gabarito)
- Após montar o snapshot, o fluxo continua normalmente em `handle_question_intake()` via `bank_match` / `student_submitted`
- `source='ocr_photo'` vai para `submitted_questions.source` (novo campo); `source_mode='ocr_photo'` vai para `QuestionSnapshot.source_mode`
