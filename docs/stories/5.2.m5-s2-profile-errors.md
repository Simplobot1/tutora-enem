## Story
**As a** estudante usando a Tutora ENEM,
**I want** digitar `/perfil` e receber um resumo dos meus pontos fracos e taxa de acerto,
**so that** eu possa entender onde errei mais, que tipo de erro cometo com mais frequência e como está minha evolução geral.

## Acceptance Criteria
1. [AC1] Handler `/perfil` registrado no `telegram_webhook.py` como fast-path (antes do pipeline me-testa), semelhante ao `/relatorio`.
2. [AC2] Query: TOP 5 tópicos com mais erros (agrupado por `topic`, contagem DESC, filtrado por `telegram_id` e `answered_correct=false`).
3. [AC3] Query: % de acerto por `final_error_type` (conceitual, interpretação, atenção) — somente registros com erro.
4. [AC4] Query: total de questões respondidas (todos os registros de `submitted_questions` para `telegram_id`).
5. [AC5] Query: taxa geral de acerto (`answered_correct=true / total`, float 0.0–1.0).
6. [AC6] Formatação Telegram: texto plano, emojis visuais (`📊 Seu Perfil`, `📉 Fraquezas`, `✅ Acertos`), máximo 2000 chars (enviar 2 mensagens se exceder).
7. [AC7] Privacidade: `ProfileService.generate(telegram_id)` valida que só retorna dados do `telegram_id` solicitante — nenhuma query sem filtro.
8. [AC8] Paginação: se usuário tem >100 questões, exibir resumo compacto (não listagem linha a linha).
9. [AC9] Testes end-to-end: 3 cenários (novo usuário sem dados, 50 questões, 500+ questões com paginação).

### Schema disponível em `submitted_questions`
Campos relevantes para as queries (migration `20260410_add_submitted_questions.sql`):
- `telegram_id BIGINT` — filtro de privacidade obrigatório em TODA query
- `answered_correct BOOLEAN` — null quando ainda não respondida; filtrar `IS NOT NULL` para total real
- `final_error_type TEXT` — valores esperados: `'conceitual'`, `'interpretacao'`, `'atencao'` (ver `app/domain/error_classification.py`)
- `topic TEXT` — tópico da questão (ex: "Funções", "Ecologia")
- `subject TEXT` — disciplina (ex: "Matemática", "Biologia")
- `created_at TIMESTAMPTZ` — disponível para filtros futuros (não necessário nesta story)

### Padrão de instanciação do Supabase no webhook
```python
# Mesmo padrão do /relatorio handler (telegram_webhook.py:48-53)
supabase_client = SupabaseClientFactory(
    url=settings.supabase_url,
    service_role_key=settings.supabase_service_role_key,
).create()
profile_service = ProfileService(supabase_client)
stats = profile_service.generate(telegram_id=telegram_id)
```

### Estrutura de retorno de `ProfileService.generate()`
```python
{
    "total": int,                          # total questões respondidas
    "acertos": int,                        # total de acertos
    "taxa_acerto": float,                  # acertos/total (0.0 se total=0)
    "top_topicos": [                       # top 5 tópicos com mais erros
        {"topic": str, "erros": int},
        ...
    ],
    "erros_por_tipo": {                    # % por tipo
        "conceitual": float,               # 0.0–100.0
        "interpretacao": float,
        "atencao": float,
    },
    "compact": bool,                       # True se total > 100
}
```

### Observações de implementação
- Queries read-only: sem necessidade de atualizar protocolo `SubmittedQuestionsRepository`; `ProfileService` usa `supabase_client` direto
- Se `supabase_client is None` (ambiente de teste sem Supabase): retornar dict vazio com mensagem informativa, não lançar exceção
- `final_error_type` pode ser `None` para questões respondidas corretamente → filtrar `IS NOT NULL` na Query 2
- Denominador de `erros_por_tipo` = total de questões com erro (não total geral)
