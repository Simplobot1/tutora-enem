## Story
**As a** operadora da Tutora,
**I want** publicar a API FastAPI em um VPS Hetzner com endpoint HTTPS permanente e deploy automatizado,
**so that** o bot do Telegram funcione de forma estavel em producao sem depender de tunnel local.

## Dependencies
- M2-S4 (Me-Testa funcional)
- Conta Hetzner ativa + acesso SSH configurado
- Domain registrado + DNS apontando para VPS
- GitHub repository com branch main

## Risks
- **Falha de DNS:** FQDN não resolve para VPS → webhook não recebe mensagens
  - Mitigation: Validar DNS antes de registrar webhook; manter IP fallback documentado
- **Expiração de TLS:** Certificado Let's Encrypt expira → conexões HTTPS falham
  - Mitigation: Configurar auto-renewal via certbot; alertar 30 dias antes
- **Webhook secret exposto:** Se `settings.webhook_secret` vazado → requisições falsificadas
  - Mitigation: Usar `SUPABASE_WEBHOOK_SECRET` em .env (nunca em git)
- **Rollback sem processo:** Deploy falha → app fica offline sem recovery
  - Mitigation: Manter último commit estável taggeado; documentar `git rollback.sh`
- **SSH auth failure:** Deploy não consegue conectar ao VPS
  - Mitigation: Testar SSH key localmente antes de setup; backup via manual push
- **Healthcheck timeout:** Pós-deploy, API não responde a `/health` → GitHub Actions falha
  - Mitigation: Aumentar timeout (30s); validar logs com `docker logs`

## Acceptance Criteria
1. A API FastAPI sobe em producao no VPS Hetzner via Docker Swarm, com container reiniciando automaticamente e endpoint de healthcheck acessivel em `GET /health`.
2. O webhook do Telegram fica exposto em URL publica permanente com HTTPS obrigatorio, usando um FQDN apontado para o VPS e reverse proxy com terminacao TLS; o endpoint publicado eh `POST /webhooks/telegram`.
3. O procedimento de registro do webhook fica documentado com comando verificavel de `setWebhook`, incluindo `secret token` quando `settings.webhook_secret` estiver configurado.
4. O deploy remoto via GitHub Actions em push para `main` usa usuario SSH definido, diretorio remoto definido, autenticacao do servidor com o repositorio definida e comandos idempotentes de update/rebuild/restart.
5. O workflow de deploy falha se healthcheck pos-deploy nao responder com sucesso e inclui procedimento minimo de rollback documentado para retornar ao ultimo release/commit estavel.
6. Os arquivos de infraestrutura necessarios para operacao ficam versionados no repositorio, no minimo `Dockerfile`, `docker-compose.yml` e `.github/workflows/deploy.yml`.
7. A story nao inclui escopo de multimodal, OCR, fotos ou novas features de produto; o escopo desta entrega eh restrito a deploy, exposicao HTTPS, webhook e CI/CD.
8. A validacao local e pre-deploy usa comandos reais do projeto, no minimo:
   - `python -m pytest`
   - `python -m pytest tests/test_health.py tests/test_models_and_webhook.py`
   - smoke da API/container com verificacao de `GET /health`
